# imessage-exporter-viewer (fork)

A self-hosted web app for searching and browsing iMessage exports generated by [imessage-exporter](https://github.com/ReagentX/imessage-exporter). Runs entirely in Docker, stores a deduplicated SQLite index, and serves a paginated chat interface with full-text search and semantic image search.

![imessage-exporter-viewer screenshot](https://raw.githubusercontent.com/amjadkamali/imessage-exporter-viewer/main/screenshot.png)

**This is a fork of [mbaran5/imessage-exporter-viewer](https://github.com/mbaran5/imessage-exporter-viewer)**, adapted to work with a source pipeline that runs `imessage-exporter -c disabled` and stores attachments in separately-named `Attachments/` and `StickerCache/` folders rather than the standard `-c clone/basic/full` layout's single lowercase `attachments/` folder. It also fixes bugs present in the upstream code and adds several UI features. See [What's Different From Upstream](#whats-different-from-upstream) for the full list.

Heads up, this project / README is mostly Claude.

## Features

- **Full-text search** across all conversations with porter stemming and prefix matching (`taco` finds `tacos`), scoped to message content only (searching a contact's name won't surface every message *sent by* them)
- **Search within a conversation** — a magnifying-glass icon in the conversation header opens an in-thread search with a match counter and prev/next navigation
- **Find a thread** — a live filter box above the conversation list matches by contact/group name or by partial phone number, so you can locate a thread even if the number hasn't been resolved to a name. Multi-word queries match each word independently, so `tom beth` finds a group named "Tom & Beth" even though that exact phrase never appears anywhere
- **Contact & group merging** — conversations spread across multiple export filenames because the same person or group used different handles are automatically recognized as one conversation. See [Contact & Group Merging](#contact--group-merging) below for how this works and its safety limits
- **Conversation Info** — an ⓘ icon in the conversation header shows exactly who a conversation resolves to (names and every known handle) and which underlying export files were merged into it, so an automatic merge decision is never a black box
- **Semantic image search** — search your photos and attachments with natural language (`"dog at the beach"`, `"birthday cake"`, `"screenshot"`) powered by [MobileCLIP-S0](https://github.com/apple/ml-mobileclip) running entirely on CPU
- **Virtual scroll** — conversations with tens of thousands of messages scroll smoothly without pagination UI
- **Jump to date** — collapsible month/year dropdowns (behind a calendar icon) jump straight to a point in a conversation's history
- **Jump to top / bottom** — one-click buttons to either end of a conversation
- **Inline reply previews** — tapbacks, threaded replies, and edited messages render faithfully from the original exporter HTML
- **Multi-archive deduplication** — index multiple export snapshots without duplicate messages
- **HEIC/HEIF support** — iPhone photos converted to JPEG on the fly for browser display
- **Name resolution** — phone numbers and emails resolved to contact names via your macOS Address Book, across group chats
- **Reply navigation** — ⇱ / ⇲ arrows jump between a reply and its parent message
- **Search result highlighting** — clicking a result jumps directly to the message with a yellow highlight

## Requirements

- [imessage-exporter](https://github.com/ReagentX/imessage-exporter) output directory (one or more export snapshots)
- Docker and Docker Compose
- CPU-only system is sufficient — no GPU required

## Quick Start

### Docker Compose

1. Clone the repository:
   ```bash
   git clone https://github.com/amjadkamali/imessage-exporter-viewer.git
   cd imessage-exporter-viewer
   ```

2. Edit `compose.yaml` to point at your export directory:
   ```yaml
   volumes:
     - /path/to/your/imessage_exports:/archives:ro
     - imessage_db:/data
   ```

3. Start the stack:
   ```bash
   docker compose up -d
   ```

4. Open [http://localhost:6333](http://localhost:6333)

The indexer runs automatically on every container start. On first run it will parse and index all exports — this may take a few minutes depending on archive size. Subsequent starts are fast as messages are deduplicated by GUID.

### Running locally

```bash
# Install PyTorch (CPU) first
pip install torch torchvision --index-url https://download.pytorch.org/whl/cpu

# Install ml-mobileclip from GitHub and remaining deps
pip install git+https://github.com/apple/ml-mobileclip.git
pip install -r requirements.txt

export ARCHIVE_ROOT=/path/to/imessage_exports
export DB_PATH=/path/to/imessage.db

python3 indexer.py          # text indexing (fast)
python3 indexer.py embed    # image embedding (slow on first run, then incremental)
python3 app.py
```

## Contact & Group Merging

`imessage-exporter` names each export file after the raw handles or the chat name it saw at export time, which means the *same* real conversation can end up split across several different filenames over time — someone's phone number changes, they start messaging from an email address, or a group name collides with an unrelated chat. The indexer resolves this by grouping conversations that genuinely represent the same people, without needing you to merge anything by hand. This runs automatically on every index; nothing needs to be enabled.

There are four cases, each handled differently depending on how much can be verified:

- **A 1:1 conversation whose handle is a known Address Book contact** merges with every other conversation that resolves to that *same* contact — so someone's old phone number's file and their email's file collapse into one thread once your Address Book has both handles on file.
- **A comma-separated group** (e.g. `+1555…, +1999….html`) is matched by its *resolved* participant set rather than its literal handles, so the same real group of people is recognized as one conversation even if a member's handle changed between exports — as long as the Address Book has both their old and new handle.
- **A named group whose filename collides** — `imessage-exporter` appends `- 2`, `- 3`, etc. to a group's filename when its custom name collides with a different internal chat identifier that happens to share the same name, which usually means the same real-world group got split into a new internal chat room (a participant change, an SMS/iMessage transition, a sync quirk). Since a name match alone can't fully rule out two genuinely unrelated groups happening to share a name, real membership data (see the [participant sidecar](#the-participant-sidecar) below) is checked as a safety net: candidates only merge if they share at least one known member. This is deliberately not a requirement that their *entire* membership match — real groups gain and lose members over time without becoming a different conversation, so demanding perfect agreement would incorrectly split an ordinary, still-active group the moment anyone's membership changed. A candidate with no membership data on file yet falls back to the plain name match, since absence of data isn't evidence the groups are different.
- **An auto-generated "guessed name" group** — when a group has no custom name, `imessage-exporter` names it something like `John, Jane & 3 others.html`, which is itself just a guess and not reliable enough to parse as real membership. These groups get the same resolved-participant-set matching as an ordinary comma-separated group, but sourced from the [participant sidecar](#the-participant-sidecar) instead of the filename — which is what lets a guessed-name file merge with another guessed-name file, or with an ordinary comma-separated file, that resolves to the exact same people.

Click the ⓘ icon in any conversation's header to see exactly what it resolved to: every participant's name and all of their known handles, plus the literal list of underlying export files that were merged together. If a merge ever looks wrong, this is where to check it.

## The Participant Sidecar

Named and auto-generated ("guessed name") group filenames carry no reliable handle information of their own — there's nothing in `Real Group Chat.html` or `John, Jane & 3 others.html` to resolve against an Address Book or match against another export. To make these groups eligible for the same merging and Conversation Info display as an ordinary group, their real membership needs to come from somewhere else: macOS's own Messages database (`chat.db`), which knows exactly who's in every chat via its `chat_handle_join` table.

If you're exporting with the companion incremental sync script [(see here)](https://github.com/amjadkamali/imessage-exporter-incremental), it takes a read-only snapshot of `chat.db` as part of each export run and writes a small sidecar file, `group_participants.json`, alongside the exported HTML — mapping each named/guessed-name file to the real list of handles `chat.db` has on record for it at that moment. The indexer reads this sidecar (via `populate_raw_participants()`) and stores it in a `conversation_raw_participants` table, which powers:

- the participant list shown in the Conversation Info modal for these groups, and
- the participant-overlap checks described above that let named-group collisions and guessed-name groups merge safely.

This is a point-in-time snapshot, not a live sync — a group's membership can genuinely change between one export and the next, and that's expected and handled: the merging logic only ever requires *some* shared membership across snapshots, never a perfect match, precisely because real groups evolve.

**If the sidecar is missing entirely** — an export produced without the incremental sync script, an older archive from before this feature existed, or a specific chat that `chat.db` had no matching record for at export time — nothing breaks. That archive's named and guessed-name groups simply fall back to being matched only by their exact filename, exactly as they would without this feature, and the Conversation Info modal omits the Participants section for them rather than showing anything misleading.

## Semantic Image Search

Clicking the **Images** button in the search bar toggles image search. Type any natural language description and the app returns matching photos and attachments ranked by visual similarity.

### How it works

1. On first container start, after text indexing completes, the image embedder runs in the background
2. Each image attachment is encoded into a 512-dimensional embedding vector using MobileCLIP-S0 and stored in the SQLite database
3. At search time the text query is encoded with the same model and compared against all stored embeddings via cosine similarity
4. Results above a 0.15 similarity threshold are returned, sorted by relevance, paginated 100 per page

### First-run timing

Image embedding is CPU-bound. Rough estimates at batch size 32:

| Collection size | Approximate time |
|---|---|
| 5,000 images | ~10 minutes |
| 20,000 images | ~45 minutes |
| 50,000 images | ~2 hours |

The web server starts immediately after text indexing completes — you do not need to wait for image embedding to finish before using the app. Image search results will appear progressively as embedding catches up. The `~30 MB` MobileCLIP-S0 model weights are downloaded from Apple's CDN on first use and cached in the `MODEL_DIR` volume.

Embedding is incremental: restarting the container only processes images that have not yet been embedded.

## Exporting your iMessages

For a one-time export, install [imessage-exporter](https://github.com/ReagentX/imessage-exporter) and run:

```bash
imessage-exporter -f html -o ~/imessage_exports
```

Point `ARCHIVE_ROOT` at the output directory. You can export multiple times over the years and point them all at the same `ARCHIVE_ROOT` subdirectory — the indexer will deduplicate across snapshots.

For ongoing, incremental exports (new messages only, each run), see the companion `imessage-incremental-sync.sh` script and its own `incremental-sync-README.md` — it also refreshes and merges your Address Book cache and writes the [participant sidecar](#the-participant-sidecar) that powers the contact-merging features above. Exports produced by plain `imessage-exporter` work fine with this viewer too; they just won't have sidecar data for named/guessed-name groups, which degrades gracefully as described above.

## Archive Layout

The indexer auto-discovers export directories. Supported layouts:

```
/archives/                          # single export at root
/archives/export-2023/              # multiple exports as subdirectories
/archives/exports/2023/             # one level deeper
```

Each discovered directory must contain at least one `.html` file and an attachment folder — either the standard lowercase `attachments/`, or the Apple standard `Attachments/` and `StickerCache/`. Any mix of these across different subdirectories is fine; each is checked independently, so you can point `ARCHIVE_ROOT` at a parent directory containing both an imessage-exporter `-c disabled` export and an older standalone `-c clone` export, and both will be discovered, indexed, and deduplicated against each other.

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `ARCHIVE_ROOT` | `/archives` | Root directory containing iMessage HTML exports |
| `DB_PATH` | `/data/imessage.db` | Path to the SQLite database |
| `MODEL_DIR` | `/data/models` | Directory where the MobileCLIP-S0 checkpoint is cached |
| `ADDRESSBOOK_CACHE_DIR` | `/addressbook-cache` | Directory holding a cached copy of your macOS Address Book (`AddressBook-v22.abcddb`, plus any `Sources/*/AddressBook-v22.abcddb` for iCloud/Exchange/On My Mac accounts), used to resolve handles to contact names. Populated and kept current by the incremental sync script; if this directory is empty or missing, name resolution and contact-based merging are skipped entirely and everything else continues to work as normal |
| `IMESSAGE_MY_HANDLES` | *(empty)* | Space-separated list of your own phone number(s) and/or email(s) (e.g. `+15551234567 me@example.com`). Used to filter out your own handle when it occasionally, incorrectly appears in a group's participant list — a known Messages glitch, not a real extra participant. Optional; leave unset if you don't need this |
| `WATCH_INTERVAL_SECONDS` | `3600` | How often (in seconds) the indexer polls `ARCHIVE_ROOT` for new or changed export files and automatically re-indexes. Only relevant if you're running new exports into the same volume while the container stays up |

## What's Different From Upstream

### Fixes

- **Timestamp parsing.** The original regex required zero whitespace between `<span class="timestamp">` and the inner `<a>` tag, but real `imessage-exporter` output always has a newline there — so every message's timestamp silently failed to parse, which showed up as conversations displaying a `? → ?` date range. Fixed by allowing whitespace in the match.
- **Sender names leaking into message search.** The full-text index covers both message content and sender name, and the original search matched against both — so searching a contact's name surfaced every message *sent by* them, not just messages *about* them. Search is now scoped to message content only.
- **`<input type="month">` has no picker UI in Safari at all** (it's simply not implemented — not a bug on this project's end). Replaced with plain `<select>` dropdowns for month and year, which render identically and correctly across browsers.
- **Messages/Images button highlight state.** The base button style defaulted to the "active" color, and the Messages button never had an explicit inactive style — so both buttons could appear highlighted at once depending on which page you were on. Rebuilt as a single consistent `.active` class applied to exactly one button at a time.

### New features

- **Search within a conversation** (magnifying-glass icon in the conversation header): live match count with prev/next navigation, reusing the same row-lookup-and-scroll mechanism that already powered global search results and reply-arrow navigation.
- **Find a thread**: a live-filtering box above the conversation list, matching by display name or by raw phone number (including partial digits), so an unresolved contact or a number buried in a group chat's filename is still findable. Multi-word queries match each word independently rather than as one literal phrase. Includes a clear button that appears once you've typed something.
- **Contact & group merging**: 1:1 conversations, comma-separated groups, named-group filename collisions, and auto-generated "guessed name" groups are all automatically recognized as the same conversation when they genuinely represent the same people — see [Contact & Group Merging](#contact--group-merging).
- **Conversation Info modal** (ⓘ icon): shows a conversation's resolved participants (with every known handle per person) and the literal underlying export files merged into it.
- **Multi-source Address Book resolution**: contact lookups merge your macOS Address Book's main store with every linked account source (iCloud, Exchange, On My Mac), not just one.
- **Jump to date** is now collapsed behind a calendar-icon toggle instead of always being visible, with its own close button.
- **Jump to top**, alongside the existing jump-to-bottom.
- **Messages/Images as a persistent mode toggle**: clicking either always updates which one is active, but only navigates to results if there's a query typed — clicking Images with an empty search box just arms that mode without leaving the current view.
- **No more image-search landing page.** Since the mode toggle never navigates without a query, the old "Enter a query to search your images" page is unreachable through normal use; direct navigation to it without a query redirects home instead.
- **No more separate "Conversations" link.** The 💬 icon in the header is now clickable and returns home from anywhere, and a clear button inside the main search box does the same after clearing the query.

## Notes

- Large conversations (40MB+ HTML files) are handled via virtual scrolling — only ~150 messages are in the DOM at a time
- HEIC images are converted to JPEG on the fly; this requires `libheif-dev` (included in the Docker image)
- The FTS index uses the SQLite porter stemmer, so text searches are stemmed and prefix-matched automatically
- Re-running the indexer is safe and idempotent — existing messages are not duplicated
- Image embeddings are stored in the same SQLite database as messages; at ~2 KB per image, 50,000 images adds roughly 100 MB to the database
- Image search quality is best for concrete subjects (`"dog"`, `"food"`, `"ocean sunset"`); abstract or text-heavy queries work less reliably
- Contact and group merging rebuild automatically on every index run, so a renamed contact or an updated Address Book cache take effect on the next restart without needing to re-export anything
